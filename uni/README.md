# 📱 QSY Uni 微信小程序前端项目

> **基于 Vue 3 + TypeScript + Vite + uni-app 的企业级小程序前端项目**  
> 本项目是全栈系统的小程序前端，包含短视频助手、用户中心、个性化主题切换、广告收益激励及全局帮助内容展示等丰富模块。

---

## 🛠️ 环境要求与本地依赖

在开始开发前，请确保您的本地开发环境已安装并配置好以下环境：

*   **运行环境**：Node.js `>= 18.0.0`
*   **包管理器**：pnpm `>= 8.0.0`
*   **预览工具**：微信开发者工具（最新稳定版）

### 🔗 本地库 Link 开发依赖

本项目为了支持核心底座的敏捷开发，强依赖于本地物理路径下的两个核心基础设施包。在执行 `pnpm install` 前，请确保以下目录关系正确建立：

*   **运行组件库**：`@hlw-uni/mp-vue` -> 指向本地物理路径 `F:/uniapp/hlw-uni/mp-vue`
*   **构建编译器**：`@hlw-uni/mp-vite-plugin` -> 指向本地物理路径 `F:/uniapp/hlw-uni/mp-vite-plugin`

## ⚙️ 常用编译指令与工作流

本项目的编译体系完美支持开发调试与生产包的高度隔离，常用指令如下：

| 命令 | 运行机制 / 读取环境 | 适用场景 |
| :--- | :--- | :--- |
| `pnpm install` | 自动配置物理 Link 依赖并安装三方包 | 初始化项目 / 依赖变更 |
| `pnpm dev` | 开发环境打包，读取 `.env.dev` 变量 | 本地实时调试与 simulator 预览 |
| `pnpm type-check` | 执行 TypeScript 严格类型与 Vue 模版语法安全检查 | 提交代码及上线前的质量保证 |
| `pnpm build` | 生产环境全量编译，读取 `.env.prod` 变量 | 体验版发布、线上版本审核上传 |

> [!WARNING]
> **线上发布与体验版上传红线：**
> 在使用微信开发者工具上传小程序版本（体验版或线上包）前，**必须且强制**先在本地执行 `pnpm build` 进行生产模式全量构建，然后再将微信开发者工具的项目目录指向 **`dist/build/mp-weixin`** 进行上传。
> **绝对禁止**直接上传 `dist/dev/mp-weixin` 开发包，否则小程序运行时会错误调用 `.env.dev` 的开发环境接口地址，导致线上数据故障。

## 🗂️ 核心目录结构划分

项目代码秉持模块化、低耦合、极简可读的原则进行组织：

```text
src/
├── common/         # 公共引导模块（如 bootstrap.ts，挂载全局变量与指令）
├── core/           # 业务组合逻辑层（Hooks / Composables，实现页面逻辑抽离）
├── service/        # 接口请求层（网络请求拦截器配置及后端 API 统一封装）
├── store/          # Pinia 状态管理中心（如主题 Preset、用户凭证状态存储）
├── pages/          # 小程序页面视图组件（按业务模块划分目录）
│   └── user/       # 用户中心模块
│       ├── comps/  # 页面专属子组件（必须以 comp-xxx.vue 命名且尽量极简）
│       └── theme/  # 主题设置页面
├── static/         # 静态资源层（包括本地图片、全局 CSS 基础变量定义）
├── types/          # 全局 TypeScript 强类型定义文件
├── App.vue         # 小程序顶级入口生命周期文件
├── main.ts         # Vue 实例初始化与插件挂载入口
└── pages.json      # 小程序页面路由与全局窗口外观配置文件
```

## 🎨 前端开发约定与代码规范

为了保障代码的高可维护性与视觉一致性，所有参与本项目的开发者或 AI 助手必须严格遵守以下代码美学规范：

### 1. 🔤 统一命名规范
*   **前后端交互字段**：所有的 API 请求参数、响应参数、数据库字段在前后端进行传输时，必须统一采用 **下划线风格（snake_case）**。例如：使用 `video_url`、`source_url`、`user_id` 等，严禁使用驼峰命名（如 `videoUrl`）。
*   **组件与函数命名极简**：组件名、变量名、方法命名必须秉持 **极简、直接** 的原则，使用含义清晰的高频简短单词。例如：用 `getUser` 代替 `fetchUserDetailInfoFromServer`，用 `status` 代替 `currentAccountStatusValue`。

### 2. 💬 全局交互约定
*   **轻提示 Toast**：页面内弹出轻量化 Toast 消息提示时，**统一且唯一**使用公共封装的方法：
    ```typescript
    hlw.$msg.toast('提示内容');
    ```
    严禁直接调用 `uni.showToast()`，以保障全局视觉样式与延迟交互的高度一致。

### 3. 📐 布局尺寸与圆角变量
*   **尺寸单位**：除极少数需要精确的极细像素边框外，编写 CSS/SCSS 样式属性时**一律统一且强制使用 `rpx` 响应式像素单位**（严禁在样式中随意混用 `px`），以确保小程序在移动端不同屏幕宽度下能自适应缩放。
*   **系统圆角与边距变量**：严禁随意使用过大的硬编码圆角与外边距数值，必须优先使用系统定义的默认变量，以保证视觉规范统一：
    *   `var(--radius-sm)` (8rpx)：适用于小标签、小按钮。
    *   `var(--radius-md)` (16rpx)：适用于常规按钮、输入框、普通卡片。
    *   `var(--radius-lg)` (24rpx)：适用于卡片内层。
    *   `var(--radius-xl)` / `var(--card-radius)` (32rpx)：适用于典型大卡片外层、弹窗容器。
    *   `var(--radius-full)` (9999rpx)：适用于胶囊按钮、头像、Pill 标签。
    *   `var(--page-padding)` (26rpx) & `var(--card-padding)` (26rpx)：系统统一内外边距标准。

### 4. ⭐ 现代化图标系统
*   图标系统采用 `@unocss/preset-icons` 进行静态编译，图标数据源自三个特定包，静态图标必须且仅使用以下前缀：
    *   **FontAwesome 6 Solid**：`i-fa6-solid-` (例如：`i-fa6-solid-gear`)
    *   **FontAwesome 6 Brands**：`i-fa6-brands-` (例如：`i-fa6-brands-weixin`)
    *   **Remix Icon**：`i-ri-` (例如：`i-ri-customer-service-line`)

## 🔄 Git 本地提交与版本号增长规范

根据项目团队开发准则，在每次执行 Git 代码提交时，必须遵循以下标准化工程工作流：

### 1. 📈 提交前版本自增机制
AI 助手或开发者在本地准备提交 Git 代码（`git commit`）之前，**必须**先检查并递增本仓库对应的版本配置文件中的版本号（如前端主项目中的 `uni/package.json` 的 `version` 字段）。
*   **版本进位规则 (`X.Y.ZZ`)**：
    *   后两位的修订号 `ZZ` 必须保持两位数字宽度（自动补零，例如从 `1.0.11` 递增为 `1.0.12`，`1.0.0` 递增为 `1.0.01`）。
    *   **进位重置**：当最后两位修订号达到 `99` 时（例如 `1.0.99`），则将次版本号 `Y` 加 1，同时将修订号 `ZZ` 重置为 `01`（例如：`1.0.99` 升级后必须是 `1.1.01`）。

### 2. 📝 强制全中文详细 Commit Message
所有的 Git 提交信息（Commit Message）**必须且强制使用极其详尽的中文说明**编写。**绝对禁止**使用纯英文或过于模糊简短的英文描述（如禁止直接使用 `fix: bug`、`feat: update`）。
*   **提交内容要求**：提交信息中必须包含**升级后的版本号**，以及**本次核心修改的具体内容和原因（精确到文件或组件）**。
*   *标准中文 Commit 示例*：
    > `修补: 前端主项目版本号顺延提升至 1.0.18 & 在 hlw-nav-bar 颜色主题下使用高光与暗影混合的立体主题色渐变，带来极其高级、奢华的导航视觉观感`

### 3. 🛑 本地提交安全红线
*   **安全红线**：AI 助手**仅限且只能在本地执行 `git commit` 进行代码和版本存档**。**绝对且严厉禁止自动执行 `git push` 或将本地改动推送到任何远程/云端仓库**！发布与推送的所有控制权必须完整且唯一保留在用户手中。
