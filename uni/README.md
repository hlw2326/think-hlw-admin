# 📱 Uni 微信小程序前端项目

> **基于 Vue 3 + TypeScript + Vite + uni-app 的企业级小程序前端项目**
> 本项目是 HlwAdmin 全栈管理系统的小程序端，包含用户中心、个性化主题切换、工具箱数据交互以及广告收益变现等核心业务模块。

---

## 🛠️ 环境要求与本地依赖

在开始开发前，请确保本地开发环境已配置以下依赖：

- **运行环境**：Node.js `>= 18.0.0`
- **包管理器**：pnpm `>= 8.0.0`
- **预览与调试**：微信开发者工具（最新稳定版）

### 🔗 本地基础设施包 Link 依赖
本项目强依赖于同级物理路径下的两个核心底座组件包。在执行 `pnpm install` 前，请确保以下目录和依赖路径正确建立：
- **核心组件库**：`@hlw-uni/mp-vue` $\rightarrow$ 指向物理路径 `F:/uniapp/hlw-uni/mp-vue`
- **构建编译器**：`@hlw-uni/mp-vite-plugin` $\rightarrow$ 指向物理路径 `F:/uniapp/hlw-uni/mp-vite-plugin`

## ⚙️ 编译指令与工作流

项目支持完整的开发环境与生产环境隔离。

| 指令 | 运行机制与环境 | 适用场景 |
| :--- | :--- | :--- |
| `pnpm install` | 自动配置物理 Link 并安装三方包 | 初始化项目或依赖发生变更时 |
| `pnpm dev` | 开发环境打包，读取 `.env.dev` | 本地实时调试与模拟器预览 |
| `pnpm type-check` | 执行 TypeScript 严格类型与 Vue 模板语法检查 | 提交代码及上线前的质量校验 |
| `pnpm build` | 生产环境全量编译，读取 `.env.prod` | 体验版发布、线上版本审核上传 |

> [!WARNING]
> **线上发布与体验版上传红线：**
> 在使用微信开发者工具上传小程序版本（体验版或线上包）前，**必须且强制**先在本地执行 `pnpm build` 进行生产模式全量构建，然后再将微信开发者工具的项目目录指向 **`dist/build/mp-weixin`** 进行上传。绝对禁止直接上传 `dist/dev/mp-weixin` 开发包，否则小程序运行时会错误调用开发环境接口地址。

## 🗂️ 核心目录结构划分

项目代码秉持模块化、低耦合、极简可读的原则进行组织：

```text
src/
├── common/         # 公共引导模块（如 bootstrap.ts，挂载全局变量与指令）
├── core/           # 业务组合逻辑层（Hooks / Composables，实现页面逻辑抽离）
├── service/        # 接口请求层（网络请求拦截器配置及后端 API 统一封装）
├── store/          # Pinia 状态管理中心（如主题 Preset、用户凭证状态存储）
├── pages/          # 小程序页面视图组件（按业务模块划分目录）
│   └── user/       # 用户中心模块
│       ├── comps/  # 页面专属子组件（必须以 comp-xxx.vue 命名且尽量极简）
│       └── theme/  # 主题设置页面
├── static/         # 静态资源层（包括本地图片、全局 CSS 基础变量定义）
├── types/          # 全局 TypeScript 强类型定义文件
├── App.vue         # 小程序顶级入口生命周期文件
├── main.ts         # Vue 实例初始化与插件挂载入口
└── pages.json      # 小程序页面路由与全局窗口外观配置文件
```

## 🎨 前端开发约定与代码规范

为了保障代码的可维护性与视觉一致性，所有参与本项目的开发者必须遵守以下代码美学规范：

### 1. 命名规范与极简原则
- **前后端交互字段**：所有的 API 请求参数、响应参数、数据库字段在前后端进行传输时，统一且强制采用 **下划线风格（snake_case）**。例如：使用 `video_url`、`source_url`、`user_id` 等，严禁使用驼峰命名（如 `videoUrl`）。
- **极简命名原则**：组件名、变量名、方法命名必须秉持极简、直接的原则，使用含义清晰的高频简短单词。例如：用 `getUser` 代替 `fetchUserDetailInfoFromServer`，用 `status` 代替 `currentAccountStatusValue`。

### 2. 全局交互与常用方法
- **轻提示 Toast**：页面内弹出轻量化 Toast 消息提示时，**统一且唯一**使用公共封装的方法：
  ```typescript
  hlw.$msg.toast("提示内容");
  ```
  严禁直接调用 `uni.showToast()`，以保障全局视觉样式与延迟交互的高度一致。
- **一键复制功能**：当页面需要提供快捷复制文本或 ID 等功能时，**统一且唯一**强制直接在标签上使用内置的 Vue 自定义复制指令 `v-copy`，例如：
  ```html
  <view class="id-wrapper" v-copy="user?.id">
      <text class="user-id">ID: {{ user?.id }}</text>
  </view>
  ```
  绝对禁止手动绑定点击事件并调用 `uni.setClipboardData` 或底层 API，以实现交互流程简化。

### 3. 布局尺寸与圆角边距变量
- **尺寸单位**：除极少数需要精确的极细像素边框外，编写 CSS/SCSS 样式属性时**一律统一且强制使用 `rpx` 响应式像素单位**（严禁在样式中混用 `px`），以确保小程序在移动端不同屏幕宽度下能自适应缩放。
- **圆角与边距变量**：默认必须优先使用系统定义的默认变量，以保证视觉规范统一：
  - `var(--radius-sm)` (8rpx)：适用于小标签、小按钮。
  - `var(--radius-md)` (16rpx)：适用于常规按钮、输入框、普通卡片。
  - `var(--radius-lg)` (24rpx)：适用于卡片内层。
  - `var(--radius-xl)` / `var(--card-radius)` (32rpx)：适用于典型大卡片外层、弹窗容器。
  - `var(--radius-full)` (9999rpx)：适用于胶囊按钮、头像、Pill 标签。
  - `var(--page-padding)` (26rpx) & `var(--card-padding)` (26rpx)：系统统一内外边距标准。

### 4. 字体排版与样式规范
- **字体大小变量**：在编写 CSS/SCSS 的 `font-size` 时，**严禁硬编码字号值**（如 `font-size: 28rpx;` 或 `font-size: 14px;`）。必须使用系统定义的 CSS/SCSS 变量：
  - **主题档**：`var(--font-xs)`、`var(--font-sm)`、`var(--font-base)`、`var(--font-md)`、`var(--font-lg)`、`var(--font-xl)`。
  - **原子档**（精确固定字号，2rpx 步进）：`var(--font-20)` 至 `var(--font-40)`。
  - **语义化排版 Token**：`var(--text-title-size)`、`var(--text-body-size)`、`var(--text-desc-size)`。
- **字重规范**：除页面主标题、醒目大数字展示等极少数核心元素可适当使用中等/中粗字重（如 `font-weight: 500;` 或 `font-weight: 600;`）外，其他绝大多数常规文本、次级标题、菜单项、辅助说明、标签等一律保持默认字重（`font-weight: 400;` 或 `normal`），呈现清爽现代的视觉风格。
- **字符间距**：文本元素编写样式时，推荐加上 `letter-spacing: 1rpx;`，微调字符间距以使字体在手机屏幕上显得更加精致。

### 5. 现代化图标系统
- 图标系统采用 `@unocss/preset-icons` 进行静态编译，图标数据源自三个特定包，静态图标必须且仅使用以下前缀：
  - **FontAwesome 6 Solid**：`i-fa6-solid-` (例如：`i-fa6-solid-gear`)
  - **FontAwesome 6 Brands**：`i-fa6-brands-` (例如：`i-fa6-brands-weixin`)
  - **Remix Icon**：`i-ri-` (例如：`i-ri-customer-service-line`)

## 📢 智能组件与广告开发规范

为了提升组件的复用率并维持极简的页面逻辑，前端深度定制了易用性广告套件，所有页面必须遵循以下开发约定：

- **Easycom 自动导入规则避让**：为了避免过于宽泛的匹配逻辑与底层核心框架组件冲突，自定义轻量级业务组件命名一律采用特定前缀，如 `^hlw-custom-(.*)`，从而实现非破坏性按需自动引入。
- **智能自适应广告组件 (`<hlw-custom-ad>`)**：为广告位展示提供一体化降级方案。在页面中只需编写 `<hlw-custom-ad type="banner" />` 或 `type="grid"`，组件内部会根据广告全局及单体模块开启配置状态，智能调配底座预设的 Unit ID 渲染，严禁在页面中硬编码或手动传递具体的 `unit-id` 字符串参数。
- **高内聚激励视频组件 (`<hlw-reward-ad>`)**：激励视频广告一律强制使用高层封装组件 `<hlw-reward-ad :unit-id="reward_unit_id" @onHandle="handleRewardAd">`。该组件完全封装了以下关键行为：
  1. 拉起前的全局 Loading 状态阻断与 8 秒防超时定时器。
  2. 用户中途退出的二级模态确认弹窗重试（“继续观看” $\rightarrow$ 二次自动递归拉起播放；“取消” $\rightarrow$ 向外发出最终失败事件）。
  3. 播放流关闭后自动在后台重新静默预加载下一个广告，实现后续展示零延迟。
  因此，业务页面只需简单处理 `@onHandle` 派发的最终播放结果。

## 🔄 Git 本地提交与版本号增长规范

在每次执行 Git 代码提交时，必须遵循以下标准化工程工作流：

### 1. 📈 提交前版本自增机制
AI 助手或开发者在本地准备提交 Git 代码（`git commit`）之前，**必须**先检查并递增本仓库对应的版本配置文件中的版本号（如前端主项目中的 `uni/package.json` 的 `version` 字段）。
- **版本进位规则 (`X.Y.ZZ`)**：
  - 后两位的修订号 `ZZ` 必须保持两位数字宽度（自动补零，例如从 `1.0.11` 递增为 `1.0.12`，`1.0.0` 递增为 `1.0.01`）。
  - **进位重置**：当最后两位修订号达到 `99` 时（例如 `1.0.99`），则将次版本号 `Y` 加 1，同时将修订号 `ZZ`重置为 `01`（例如：`1.0.99` 升级后必须是 `1.1.01`）。

### 2. 📝 强制全中文详细 Commit Message
所有的 Git 提交信息（Commit Message）**必须且强制使用极其详尽的中文说明**编写。绝对禁止使用纯英文或过于模糊简短的英文描述（如禁止直接使用 `fix: bug`、`feat: update`）。
- **提交内容要求**：提交信息中必须包含**升级后的版本号**，以及**本次核心修改的具体内容和原因（精确到文件或组件）**。
- *标准中文 Commit 示例*：
  > `修补: 前端主项目版本号顺延提升至 1.0.18 & 在 hlw-nav-bar 颜色主题下使用高光与暗影混合的立体主题色渐变，带来极其高级、奢华的导航视觉观感`

### 3. 🛑 本地提交安全红线
- **安全红线**：仅限在本地执行 `git commit` 进行代码 and 版本存档。**绝对且严厉禁止自动执行 `git push` 或将本地改动推送到任何远程/云端仓库**。发布与推送的所有控制权必须完整且唯一保留在用户手中。
