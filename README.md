# HlwAdmin 全栈管理系统与微信小程序工程

本项目是一个前后端分离的高性能全栈系统工程，包含基于 ThinkPHP 8.2 + ThinkLibrary 6.1 的后端管理系统与基于 Vue 3 + TypeScript + uni-app 的微信小程序前端。系统设计兼顾了多小程序租户管理、高效广告收益变现与严格的代码美学标准。

---

## 📂 项目架构与目录结构

### 1. 全局架构

项目在根目录下分为后端服务与前端小程序两部分：

- `php/`：后端项目目录，负责业务逻辑、API 提供、数据库存储及系统管理。
- `uni/`：前端项目目录，负责小程序端渲染、用户交互、主题定制及微信广告变现。

```text
hlw-admin/ (项目根目录)
├── php/             # 后端项目 (基于 ThinkPHP 8.2 + ThinkLibrary 6.1)
└── uni/             # 前端项目 (基于 Vue 3 + uni-app + Vite)
```

### 2. 后端核心目录 (`/php`)

- `app/admin/`：系统后台管理模块（包含权限控制、管理员管理、系统配置等）。
- `app/wechat/`：微信接口模块（用于小程序一键登录、敏感数据解密、支付等微信开放平台业务）。
- `app/index/`：前台入口及公共基础模块。
- `package/`：本地插件目录，用于模块化扩展业务（当前内置插件包含 `think-plugin-test`）。
- `database/`：存放数据库迁移脚本（Migrations）与 SQLite 本地数据库文件。
- `public/`：Web 访问根目录，所有外部 HTTP 请求与站点部署均需指向此目录。

### 3. 前端核心目录 (`/uni/src`)

- `common/`：系统引导程序与全局注册挂载点（如全局方法挂载与复制指令）。
- `core/`：可复用的业务组合式函数层（Hooks / Composables）。
- `service/`：API 统一请求配置、接口拦截器及 API 调用封装。
- `store/`：基于 Pinia 的全局状态管理，如用户登录凭证、系统主题 Preset。
- `pages/`：视图页面，子组件强制存储于对应页面目录下的 `comps/` 文件夹中并使用 `comp-xxx.vue` 命名。
- `static/`：本地图片、SCSS 样式变量等静态资产。
- `types/`：全局 TypeScript 强类型定义文件。

## ⚙️ 快速开始

### 1. 后端部署步骤

```bash
cd php
composer install
cp .env.example .env     # 配置您的数据库连接与微信小程序 AppID/AppSecret 参数
php think migrate:run
php think xadmin:publish --migrate
php think run --host 192.168.5.12 --port 8000
```

> [!NOTE]
>
> - 部署至宝塔或生产环境时，请确保 `runtime/` 和 `public/upload/` 目录拥有写入权限。
> - 如果在安装依赖时发生 `runtime/cache` 写入权限错误，可运行 `composer install --no-scripts`。

### 2. 前端部署与调试步骤

```bash
cd uni
pnpm install
pnpm dev       # 启动开发环境，读取 .env.dev 变量并开始编译
pnpm build     # 构建生产环境，读取 .env.prod 变量进行代码打包优化
```

> [!WARNING]
> 在向微信小程序平台上传体验版或线上版前，**必须且强制**在本地执行 `pnpm build` 进行生产模式构建，再将微信开发者工具的项目目录指向 **`dist/build/mp-weixin`** 进行上传。禁止直接上传 `dist/dev/mp-weixin` 目录，否则小程序将错误连接至开发环境接口。

## 💎 系统核心设计与技术亮点

### 1. 动态多小程序租户绑定

- **架构设计**：`test_user` 表引入了 `appid` 存储，微信一键登录与注册流程能根据用户来源动态追踪。
- **业务场景**：实现单套系统后台能够为多个不同的小程序租户提供统一的管理与数据分析支持。

### 2. 精准工具箱热度监控

- **功能描述**：`test_tools` 数据表引入 `click_count` 热度分析字段，配合后台 `Tools::click` 接口。
- **业务价值**：前端用户在小程序中点击工具卡片时实时递增点击数，为管理运营提供实时精确的模块点击率统计。

### 3. 高内聚自适应广告套件

- **智能广告位 `<hlw-custom-ad>`**：支持内置 fallback 和广告位自动路由，仅需传入广告类型参数（如 `type="banner"` 或 `type="grid"`），无需在页面中硬编码 Unit ID 字符串。
- **激励视频组件 `<hlw-reward-ad>`**：
  - 具备 8 秒加载防超时阻断与全局 Loading 控制。
  - 内置中途退出二级模态确认弹窗，引导用户点击“继续观看”继续播放或“取消”退出。
  - 支持播放关闭后，在后台静默预加载下一个激励广告，降低后续拉起延时。

### 4. 纯净主题变量样式系统

- **排版与大小**：全局使用 `rpx` 作为尺寸和边距单位，提供统一的圆角变量（`--radius-sm/md/lg/xl/full`）与内边距变量（`--page-padding` / `--card-padding`）。
- **字体标准**：禁止硬编码字号，统一使用系统字体大小变量（如 `var(--font-sm)`, `var(--font-md)`）；推荐文本样式中微调字符间距（`letter-spacing: 1rpx;`）提升视觉阅读质感。
- **无 UnoCSS 堆叠**：禁止在 HTML 标签上大量堆叠原子化样式类，所有业务组件外观定义集中编写在 `<style scoped lang="scss">` 或 SCSS 文件中，维持代码整洁。

## 🤝 开发协作与规范约束

为了维护代码一致性，所有参与开发的成员（包含 AI 编码助手）必须严格遵循以下开发标准：

### 1. 前后端接口字段格式

- 所有 API 请求参数、响应参数、数据库字段在数据交互时，**统一且唯一**使用 **下划线格式（snake_case）**。例如：使用 `video_url`、`source_url`。

### 2. 前端快捷指令与方法

- **信息弹窗**：轻量提示统一使用 `hlw.$msg.toast('提示内容')`，禁止直接调用 `uni.showToast()`。
- **文本复制**：复制文本统一在元素上配置 `v-copy` 自定义指令（如 `v-copy="user.id"`），禁止手动编写复制点击事件。

### 3. Git 版本自增与 Commit 机制

- **版本号格式**：版本号格式遵循 `主版本号.次版本号.修订号`（即 `X.Y.ZZ`），修订号 `ZZ` 必须补零保持两位数（如 `1.0.12`）。
- **提交前自增**：在本地准备执行 `git commit` 前，**必须**先检查并递增本仓库对应的版本配置文件中的版本号（前端为 `uni/package.json`，PHP 插件为各自的 `composer.json`）。
- **进位规则**：当修订号达到 `99` 时（如 `1.0.99`），再次升级版本时必须进位至次版本号，并将修订号重置为 `01`（如 `1.1.01`）。
- **Commit 格式**：提交说明必须且强制使用极其详尽的中文编写，必须包含本次修改的组件或模块，以及变动细节与升级后的版本号。
- **安全红线**：AI 编码助手仅允许在本地执行 `git commit` 进行归档，**绝对禁止自动执行 `git push` 命令**。
